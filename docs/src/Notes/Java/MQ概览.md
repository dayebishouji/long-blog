## 消息队列概览

消息队列（MQ）是一种异步通信的中间件，根据消费模型的不同，主要分为两大类。

---

## 一、消息队列的两大模型

### 1. AMQP模型（点对点 / 竞争消费）

**特点：**
- 允许多个消费者共同消费同一个队列
- 消息处理和确认是**破坏性操作**（消息被消费后即被删除）
- 各消费者竞争同一份消息
- 适合任务分发场景

**代表产品：RabbitMQ**

![](/minio/weblog/b5c2ba9dbfda467bb57cf17736999e28.png)

**工作流程：**
```
Producer → Queue ← 多个Consumer（竞争消费）
          消息被消费后自动删除
```

**典型应用：**
- 📧 邮件发送队列
- 🔧 任务分发系统
- 📝 订单处理队列

---

### 2. 日志模型（发布订阅 / 重复消费）

**特点：**
- 消息不会被消费后删除，而是持久化存储
- 支持多个消费者**各自独立**地消费（从不同的偏移量开始）
- 消费是非破坏性操作
- 适合日志流、事件流场景

**代表产品：Kafka、RocketMQ**

![](/minio/weblog/35697757d9f347949df14b5ae05054ff.png)

## 二、Kafka深度解析

### 高吞吐设计

Kafka能达到**百万级/秒**的吞吐量，主要得益于以下设计：

#### 1. 双线程模型

```
┌─────────────────────────────────┐
│   Producer应用代码                │
└──────────────┬──────────────────┘
               │
        ┌──────▼──────┐
        │   主线程     │
        ├──────────────┤
        │ • 创建消息   │
        │ • 调用拦截器 │
        │ • 序列化消息 │
        │ • 放入累加器  │
        └──────┬───────┘
               │
        ┌──────▼──────────────────┐
        │   RecordAccumulator      │
        │   (内存缓冲区)            │
        └──────┬───────────────────┘
               │
        ┌──────▼──────┐
        │  Sender线程  │
        ├──────────────┤
        │ • 获取批次消息│
        │ • 压缩处理    │
        │ • 发送到Broker│
        │ (16KB/1s)   │
        └──────┬───────┘
               │
            Broker
```

**作用：**
- ✅ 主线程专注于消息创建，不阻塞业务
- ✅ Sender线程负责批量发送，提高吞吐量
- ✅ 批次大小：16KB 或 1秒超时（取决于哪个先满足）

#### 2. 分区设计

不同分区消息的分配策略：

| 策略 | 特点 | 适用场景 |
|------|------|---------|
| **轮询** | Round Robin，消息均匀分布 | 通用场景，流量均衡 |
| **粘性** | 尽量往一个分区写，分区满后切换 | 减少批次拆分，提高吞吐 |
| **Hash** | 根据key的hash分配到分区 | 相同key的消息保证顺序性 |

```java
// Kafka分区示例
Properties props = new Properties();
props.put("partitioner.class", "org.apache.kafka.clients.producer.internals.DefaultPartitioner");
// 使用Hash分区器，相同key的消息会发送到同一分区
producer.send(new ProducerRecord<>("topic", key, value));
```

#### 3. 零拷贝技术

**传统数据传输：**
```
应用缓冲 → 内核缓冲 → Socket缓冲 → 网络 (多次复制)
```

**零拷贝（sendfile）：**
```
应用缓冲 → 内核缓冲 ──┐
                    ├──→ 网络 (减少复制次数)
                Socket缓冲 ──┘
```

**优势：** 减少CPU参与，降低上下文切换

#### 4. 顺序写

```
随机写：磁盘工作时间 = 寻道时间 + 旋转延迟 + 传输时间 (ms级，很慢!!)
顺序写：直接追加写入磁盘末尾，充分利用磁盘带宽
```

Kafka通过**追加写**（append-only）完全避免随机写。

#### 5. 消息压缩

**压缩算法对比：**

| 算法 | 压缩率 | 时间 | 适用 |
|------|--------|------|------|
| None | 0% | - | 实时性要求高 |
| GZIP | 69% | 慢 | 长期存储 |
| Snappy | 49% | 快 | 通用、推荐 |
| LZ4 | 37% | 很快 | 高吞吐 |

```java
// Kafka生产者配置压缩
properties.put("compression.type", "snappy"); // 推荐
```

### Kafka的消费模型

```
┌──────────────────────────────┐
│   Topic (主题)                │
├──────────────────────────────┤
│ Partition 0  │ Partition 1   │
│ [msg0][msg1] │ [msg2][msg3]  │
└──────────────────────────────┘
       ▲              ▲
       │              │
    ┌──┴─┐         ┌──┴──┐
    │Consumer1    │Consumer2
    │ offset:2    │ offset:1
    └─────┘       └──────┘
```

**特点：**
- 每个Consumer维护自己的**offset**（消费位置）
- 同一Topic的不同Partition可被不同Consumer并行消费
- 支持**重新消费**（通过重置offset）

## 三、RocketMQ深度解析

### 架构设计

![](/minio/weblog/f00eb8ede3f549af903d3ca5cb2c0140.png)

![](/minio/weblog/6dd16e58db8e4f85b1a27434f5d9e4da.png)

**核心组件：**

| 组件 | 职责 | 说明 |
|------|------|------|
| **NameServer** | 注册中心 | 管理Broker信息，轻量级无状态 |
| **Broker** | 消息存储 | 存储消息、处理消费者请求 |
| **Producer** | 生产者 | 发送消息，支持同步/异步/单向 |
| **Consumer** | 消费者 | 消费消息，支持推/拉模式 |

### RocketMQ的特性

#### 1. 高可用性

```
多Master架构:
├─ Master Broker 1 ──┐
├─ Master Broker 2   ├─ 互相备份
├─ Master Broker 3 ──┘
       │
       └─ Slave Broker (同步副本)
```

- ✅ Master挂掉不影响消费（Consumer可从Slave拉取）
- ✅ 生产者可自动重试到其他Master
- ✅ 支持同步双写/异步双写

#### 2. 消息顺序性

RocketMQ支持**严格的顺序消费**：

```java
// 同一个MessageQueue的消息能保证顺序
// 实现原理：同一个key的消息发送到同一个Queue
DefaultMQProducer producer = new DefaultMQProducer("test_group");
producer.start();

Message msg = new Message("orderTopic", "tag", "key_001", 
    "订单消息".getBytes());
producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        int orderId = (int)arg;
        return mqs.get(orderId % mqs.size());
    }
}, 123); // 按orderId确定Queue，相同orderId的消息进同一个Queue
```

#### 3. 事务消息

```
┌─────────────┐
│ Producer    │
└──────┬──────┘
       │ 1.发送Half消息
       ↓
┌──────────────────┐
│ RocketMQ Broker  │
│ (Half Topic)     │
└──────┬───────────┘
       │ 2.执行本地事务
       ↓
┌─────────────────────────┐
│ 业务DB事务           │
│ + RocketMQ事务状态   │
└──────┬──────────────────┘
       │ 3.提交/回滚事务
       ↓
┌──────────────────┐
│ RocketMQ Broker  │
│ (消费者可见)      │
└──────────────────┘
```

**用途：** 保证消息发送与本地数据库事务的一致性

#### 4. 消费重试机制

```
消费失败 → 自动重试
    │
    ├─ 第1次：10秒后
    ├─ 第2次：30秒后
    ├─ 第3次：1分钟后
    └─ ... → 最多重试16次
    
失败16次后 → 进入死信队列(DLQ)
```

## 四、消息队列对比

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|---------|-------|----------|
| **吞吐量** | 万级 | 百万级⭐ | 十万级 |
| **消费模型** | AMQP竞争 | 发布订阅 | 发布订阅+ |
| **消息可靠** | 高 | 高 | 高⭐ |
| **顺序性** | 弱 | 强⭐ | 强⭐ |
| **事务支持** | 否 | 否 | 是⭐ |
| **延迟队列** | 否 | 否 | 是⭐ |
| **死信队列** | 是 | 否 | 是 |
| **复杂度** | 低 | 中 | 高 |
| **学习成本** | 低 | 中 | 高 |

## 五、适用场景选择

### 📧 RabbitMQ

**场景：** 任务队列、邮件发送、消息通知

```
优先选择原因：
✅ 消息必须被完整消费（无需重复）
✅ 消息优先级不同
✅ 需要复杂的路由规则
✅ 团队对RabbitMQ熟悉
```

### 📊 Kafka

**场景：** 日志采集、事件流、数据分析、流处理

```
优先选择原因：
✅ 海量日志/事件需要采集
✅ 需要消息重复消费（多个消费者独立处理）
✅ 吞吐量要求超高（百万级/s）
✅ 需要长期保留消息历史
✅ 中台/数据架构
```

**Kafka应用案例：**
```
用户事件 → Kafka Topic: user_events
    ├─ Consumer1: 写入实时数据库
    ├─ Consumer2: 推送给推荐系统
    ├─ Consumer3: 离线分析ETL
    └─ Consumer4: 监控告警系统
```

### 🚀 RocketMQ

**场景：** 订单系统、支付系统、库存同步、交易相关

```
优先选择原因：
✅ 需要严格的消息顺序性
✅ 需要事务消息保证一致性
✅ 需要消息可靠性非常高
✅ 需要大量重试机制
✅ 业务中有延迟任务需求
```

**RocketMQ应用案例：**
```
订单流程：
①创建订单(本地事务) 
    ↓
②发送【订单创建】事务消息
    ↓
③消费消息→更新库存(消息顺序保证同一订单)
    ↓
④消费消息→更新积分
```

## 七、常见面试题

### Q: 如何确保消息不丢失？

**生产端：**
- ✅ 同步发送确认 + 重试机制
- ✅ 本地消息表 + 定时扫描

**Broker端：**
- ✅ 同步双写或异步多副本
- ✅ 定期刷盘

**消费端：**
- ✅ 先消费后commit offset
- ✅ 本地事务确保幂等性

### Q: 如何处理消息重复？

```
本质：同一条消息被消费多次

解决方案：
1️⃣ 业务幂等设计
   - 例：支付操作使用支付单据ID确保不重复扣款

2️⃣ 消费端去重
   - 使用Redis/数据库记录已消费消息ID
   - 查询是否存在，存在则跳过

3️⃣ 消息ID + 业务ID组合
   - 保证同一条业务消息只处理一次
```

### Q: 消费端如何保证顺序？

```
Kafka: 
- 单线程消费 + 同一分区
- 或者保存消息顺序到优先级队列本地处理

RocketMQ:
- 自动支持，相同key的消息进同一MessageQueue
- 消费时单线程执行同一Queue
```
